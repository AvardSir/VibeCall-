Host

The person who starts the call from the landing page, invites others with the participant link, and owns the room: copy link, share screen, remove a guest, end the call. Theme (Dark default / Light) and language (EN default / RU) toggles sit top-right on every screen.

H1

Landing — Start a call

PRD: FR-30, US-17, US-1

The app's root screen. App name/logo, one-line tagline, a single "Start a call" button, plus theme and language controls. No other navigation. Clicking "Start a call" creates a room and sends the host to the pre-join screen (H2).

Chat rules
  • New joiner sees history accrued while the room has been alive; cleared when the room ends
  • Send on click or Enter; enabled when there is text or ≥1 attachment; max 1000 chars (counter appears at 900)
  • Hidden panel + new message → unread dot on chat button; cleared on open
  • Sending in progress → message shows Sending… until delivered; on failure → Not delivered, text/attachments kept to resend
Attachments & empty state
  • Images: PNG/JPEG/GIF/WebP — thumbnail; click opens a full-size overlay (dimmed bg, not fully black; close via ✕ / Esc / click outside; view only). Animated GIF/WebP are still in the chat and animate only in the overlay
  • Files: PDF/DOC/DOCX/XLS/XLSX/TXT/ZIP — chip (name + size + download); click downloads via browser, no in-app preview
  • ≤ 10 MB/file, ≤ 5/message. Errors: Unsupported file type. / File exceeds 10 MB. / You can attach up to 5 files per message.
  • No messages yet → No messages yet.
H5

In-call — screen sharing

PRD: FR-16, US-8 · any participant, one at a time

Any participant can share, one share at a time. While active, the shared content is the large main area for everyone — including the sharer, who sees their own shared content there too — and all participant videos become a thumbnail strip. The sharer's camera keeps running (two streams: screen + camera), so their camera tile stays in the strip with everyone else's. The main area is labelled — others see «<name> is sharing their screen», the sharer sees «You are sharing your screen»; every thumbnail keeps its name label and camera-off / mute indicators. The sharer gets a "Stop sharing" control; everyone else's "Share screen" button is disabled.

Rules
  • One active share at a time; others' control disabled with tooltip Someone is already sharing their screen
  • Concurrent attempts → first registered by server wins; the other is rejected
  • Sharer leaves / is removed / host grace begins → share ends, layout returns to grid
Streams & labels
  • Shared content fills the main area for everyone, incl. the sharer; main area labelled <name> is sharing their screen (You are sharing your screen for the sharer)
  • Shared screen shown in full (contain) — never cropped; neutral margins if proportions differ. (Tiles use cover; the shared screen uses contain.)
  • Sharer's camera keeps transmitting → 2 streams (screen + camera); their camera tile stays in the strip
  • Every thumbnail keeps its name label + camera-off / mute icons, in grid order (host → guests by join order)
States
  • Prompt cancelled / capture denied → inline above bar: Unable to share your screen. Please check your browser permissions. (4s)
  • System / tab audio is not captured (out of scope)
  • Bottom controls follow the viewer's role (guest: Leave, no Copy link / End)
H6

Remove a guest (host)

PRD: FR-6, US-13

The host sees a "Remove" control on each guest's tile (revealed on hover or keyboard focus). Activating it opens a confirmation dialog — "Remove <name> from the call?" with a red "Remove" and a "Cancel" — and the guest is disconnected only after the host confirms. On confirm the slot is freed and the removed guest sees screen G4; "Cancel" closes the dialog with no change. Removal does not block them from rejoining via the link.

Behaviour
  • "Remove" shown only to the host, on each guest's tile (revealed on hover/focus; tooltip Remove this guest)
  • Click "Remove" → confirmation dialog Remove <name> from the call? with red "Remove" + "Cancel"
  • Guest removed only after confirm; "Cancel" closes the dialog, nothing changes
  • Target guest leaves/disconnects while dialog open → dialog auto-closes, nothing to remove
  • On confirm: guest disconnected; grid re-arranges; slot freed
Result for the guest
  • Guest sees G4: You were removed from the call by the host. + "Back to home"
  • Guest may rejoin via the link (no rejoin block)

Guest

Anyone who opens the participant link. They join by entering a name, take part in the call (camera, mic, screen share, chat), and can leave at any time. Guests never see "Copy link", "End call", or "Remove". The chat panel (H4) and screen-sharing (H5) layouts are the same for guests — only the bottom controls differ (guest: Leave, no Copy link and no End call).

G6

Host disconnected — reconnecting

PRD: FR-4, US-14

If the host drops unexpectedly, the room enters a 60-second grace period. Guests see an overlay with a live countdown over the call. If the host returns, the call resumes; if 60s pass, the room ends.
